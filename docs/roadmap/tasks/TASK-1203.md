## Context

`src-tauri/src/commands/publish/preflight/access.rs` 在 macOS 上对 `Desktop/Documents/Downloads` 三类受保护目录做写入探针，其他平台返回 `not_applicable`。问题：

- 已挂载远程卷（如 `/Volumes/build-share`、`/mnt/nas`）不在保护目录列表里，但探针逻辑也没有给它"远程上下文"——前端无法在 UI 上区分"是本地受保护目录"还是"已挂载远程卷"。
- 未来 `Remote` 分支需要把"远端摘要"（host / path / scheme）暴露给前端，但当前 `PublishOutputAccess` 只有 `protected_location / protected_root / probe_directory`，没有位置存。
- `delete_existing_files` 的探针对远端语义不安全：默认对本地卷直接 `create_dir/remove_dir`，远端必须有显式确认（Phase 13）。R0 阶段需先把"远程上下文"暴露出来，否则 Phase 13 没法在 UI 弹窗里区分。

## Goals

- 在 `PublishOutputAccess` 上新增 `remote_location: Option<RemoteLocationSummary>` 字段（向后兼容序列化，老字段保留）
- preflight 对 `MountedRemote` 直接产出 `Granted` + `remote_location` 摘要，跳过受保护目录写入探针
- 对 `Remote(RemoteUri)` 产出 `Skipped` + `remote_location`，并附带 `detail = "remote_probe_pending_phase_r1"` 占位
- 不引入任何网络 IO

## Non-Goals

- 远程可达性/鉴权探针（Phase 13 / R1）
- `delete_existing_files` 二次确认弹窗（前端在 TASK-1204 配合做"展示"，真正的确认 + 清理在 R1）

## User Stories

- 作为 macOS 用户，我把输出目录设为 `/Volumes/build-share/publish` 后，preflight 不再做"是不是 Downloads/Documents"的扫描，结果显示"已挂载远程卷"。
- 作为后续 Phase 13 的开发者，我可以从 `PublishOutputAccess.remote_location` 直接拿到 host/path 摘要，不需要再次解析字符串。

## Requirements

### Functional

- R1: 新增结构

  ```rust
  #[derive(Debug, Clone, PartialEq, Eq, Serialize, TS)]
  #[serde(rename_all = "camelCase")]
  #[ts(rename_all = "camelCase")]
  pub struct RemoteLocationSummary {
      pub kind: RemoteLocationKind,         // "unc" | "mounted" | "remote"
      pub display: String,                  // 脱敏后的展示文本，例如 "\\\\nas01\\releases\\app"
      pub host: Option<String>,             // mounted/remote 才有
      pub scheme: Option<String>,           // remote 才有
      pub fs_type: Option<String>,          // mounted 才有
  }

  pub enum RemoteLocationKind { Unc, Mounted, Remote }
  ```

- R2: `PublishOutputAccess` 增字段 `remote_location: Option<RemoteLocationSummary>`，保留现有 `protected_location / protected_root / probe_directory`，以保证现有前端兼容。
- R3: `preflight_publish_output` 流程：
  1. 解析 `OutputTarget`；
  2. 路径合法性校验（TASK-1202 已覆盖）；
  3. 分发：
     - `Local` → 走现有 `evaluate_publish_output_access`；
     - `MountedRemote` → 不调 probe，直接返回 `Granted` + `remote_location`；
     - `Remote` → 返回 `Skipped` + `remote_location` + `detail = "remote_probe_pending"`。
- R4: 新增 `PublishOutputAccessStatus::RemoteSkipped` 子状态吗？**不新增**——用现有 `Skipped` + `detail` 字段区分，保持前端兼容。前端在 TASK-1204 内通过 `remote_location.is_some()` 区分 UI 文案。
- R5: 脱敏：`RemoteLocationSummary.display` 使用与 `security.rs` 一致的占位策略雏形——R0 阶段仅保留 host 与 path，不输出用户名/密码（即使 URI 里带也忽略）。

### Non-Functional

- N1: 新字段 `Option<>` 默认 `None`，旧前端反序列化不报错（Tauri/ts-rs 默认行为）
- N2: 零网络 IO，零 fs 系统调用（mounted 分支不读 fs_type 也行，复用 TASK-1201 的解析结果）
- N3: 前端 `bindings/PublishOutputAccess.ts` 由 `ts-rs` 重新生成，PR 中包含更新

## UX Notes

- 状态语义（前端解释，本任务只产出数据）：
  - `Granted + remote_location.kind == "unc"/"mounted"` → 显示「目标位于已挂载远程卷，跳过本地写入探针」
  - `Skipped + remote_location.kind == "remote"` → 显示「目标为远程协议，将在发布时上传」(Phase 13 才真上传)
  - `Granted` 无 `remote_location` → 保持现有"本地写入探针通过"文案

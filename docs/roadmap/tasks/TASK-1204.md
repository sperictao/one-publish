## Context

前端 PublishConfigPanel 当前输出目录输入框只是裸文本框，用户配置 UNC / 挂载远程卷 / `sftp://` 时缺乏视觉反馈：

- 不知道应用是否识别为"远程"
- preflight 弹出"远端跳过"也没有上下文来源
- 后续 Phase 13 的凭据选择器需要一个挂载点来扩展

本任务把 TASK-1201 的解析结果与 TASK-1203 的 `remote_location` 串到 UI，提供最小可识别的"目标徽章"。

## Goals

- 输出目录输入框旁渲染目标类型徽章：`本地 / UNC / 挂载远程 / 远程协议(sftp/s3/...)`
- 徽章下方显示简短摘要：host、fs_type、scheme（按类型可见性）
- preflight 卡片同步显示「跳过本地写入探针，原因：已挂载远程卷 / 远程协议待 Phase 13」

## Non-Goals

- 凭据选择器（Phase 13）
- 远程目录浏览 / 自动补全（Phase 13）
- 真正的远端可达性探针展示（Phase 13）

## User Stories

- 作为发布到 NAS 的开发者，我希望输入 `\\nas01\releases\app` 后立即在 UI 看到「UNC 共享」徽章，确认应用识别正确。
- 作为 macOS 用户，我希望输入 `/Volumes/build-share/publish` 后看到「已挂载远程卷 (smbfs)」徽章，并且 preflight 卡片说明跳过桌面保护目录探针。

## Requirements

### Functional

- R1: 新增前端工具 `src/lib/outputTarget.ts`：
  - `parseOutputTargetLocal(raw: string): OutputTargetHint`（同步、纯字符串规则，不走 IPC，用于即时渲染）
  - `parseOutputTargetRemote(raw: string): Promise<OutputTargetDescriptor>`（走 TASK-1201 暴露的 IPC，含 fs_type）
  - `OutputTargetHint = { kind: "local" | "unc" | "scheme", scheme?: string }`
- R2: PublishConfigPanel 中 `output` 输入框的 `onChange` 同步触发 `parseOutputTargetLocal`，渲染基础徽章；防抖 300ms 后调一次 `parseOutputTargetRemote` 补 fs_type、host。
- R3: 徽章组件 `OutputTargetBadge`：
  - `kind: "local"` → 文案"本地"，使用 muted 色
  - `kind: "unc"` → "UNC 共享"，蓝色
  - `kind: "mounted"` → "挂载远程卷"，蓝色 + `fs_type` 副标签
  - `kind: "remote"` → `${scheme.toUpperCase()}`，红色边框 + tooltip "远程协议将在 Phase 13 支持上传"
- R4: preflight 卡片：
  - 收到 `access.remoteLocation` 时优先展示远程摘要，覆盖原"本地探针通过"文案
  - 当 `access.status === "skipped"` 且 `remoteLocation.kind === "remote"` 时显示明确的「待 Phase 13」提示，并加 disabled 状态的"立即测试"按钮（先占位，TASK-1303 实现）

### Non-Functional

- N1: 同步徽章渲染零网络，零 IPC；IPC 调用全程不阻塞输入框
- N2: 中英文文案统一走 `src/i18n/`（如果仓库已有 i18n key 体系；如无则使用现有硬编码模式）
- N3: 在 100ms 内完成徽章首次渲染（基于本地规则）

## UX Notes

- 输入框右侧 inline 渲染徽章，避免抢主操作按钮注意力
- 徽章 hover 显示完整 tooltip：解析后的 host/path/scheme/fs_type
- 当目标为 `remote` 时输入框边框加浅色提示，避免用户误以为 Phase R0 就能上传
